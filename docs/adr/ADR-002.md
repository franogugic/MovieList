# ADR-002: Sigurnosne mjere web aplikacije

**Datum:** 2026-06-30  
**Status:** Prihvaćeno  
**Autor:** Frano Gugić

---

## Kontekst

MovieList je web aplikacija s REST API-jem i JWT autentikacijom. U ovom koraku implementirane su sigurnosne mjere prema zahtjevima kolegija "Napredni razvoj web aplikacija".

---

## Odluke i implementirane mjere

### 1. Prepared Statements (SQL Injection zaštita)

**Odluka:** Svi SQL upiti koriste PDO prepared statements s parametriziranim bindanjem vrijednosti.

**Razlog:** Direktno umetanje korisničkih podataka u SQL upit omogućuje SQL Injection napad kojim napadač može čitati, mijenjati ili brisati podatke iz baze. PDO prepared statements odvajaju SQL kod od podataka — baza prima upit i parametre odvojeno, pa zlonamjerni unos poput `' OR 1=1 --` nikad ne može izmijeniti strukturu upita.

**Primjer:** `$stmt = $pdo->prepare("SELECT * FROM users WHERE email = ?"); $stmt->execute([$email]);`

---

### 2. XSS zaštita (Cross-Site Scripting)

**Odluka:** Koriste se dvije razine XSS zaštite:

- **PHP sloj:** `htmlspecialchars()` za sve korisničke podatke koji se ispisuju u PHP view predlošcima (naslovi filmova, opisi, imena redatelja).
- **JavaScript sloj:** `escapeHtml()` funkcija koja koristi `div.textContent` za sigurno renderiranje komentara i korisničkih podataka učitanih putem Fetch API-ja.

**Razlog:** Bez escapiranja, napadač može unijeti `<script>alert(document.cookie)</script>` kao komentar koji se zatim izvršava u preglednicima svih korisnika koji pregledaju stranicu (Reflected/Stored XSS). `htmlspecialchars()` pretvara posebne znakove (`<`, `>`, `"`, `&`) u HTML entitete koji se prikazuju kao tekst, a ne kao kod.

---

### 3. CSRF zaštita (Cross-Site Request Forgery)

**Odluka:** Implementiran `Csrf` helper koji generira kriptografski sigurni token pohranjen u PHP sessioni (`random_bytes(32)`). Token se ugrađuje u login i register stranice kao JavaScript varijabla i šalje kao `X-CSRF-Token` header pri svakom Fetch pozivu. Server validira token putem `hash_equals()` za zaštitu od timing napada.

**Razlog:** CSRF napad prisiljava autenticiranog korisnika da nesvjesno izvrši neželjenu radnju. Primjer: zlonamjerna stranica šalje skriveni POST na `movielist/api/auth/login`. Budući da preglednik automatski šalje cookije, bez CSRF zaštite server bi prihvatio zahtjev kao legitiman.

**Napomena o API rutama:** Mutacijske API rute (dodavanje/izmjena/brisanje filmova na listi) zaštićene su JWT tokenom koji se šalje kao `Authorization: Bearer` header. Budući da preglednik nikada automatski ne šalje `localStorage` vrijednosti cross-origin, ove rute su inherentno zaštićene od CSRF-a bez dodatnog tokena.

---

### 4. Server-side validacija ulaznih podataka

**Odluka:** Sve ulazne vrijednosti za korisničke zapise validiraju se na serveru:

| Polje      | Pravilo                              |
|------------|--------------------------------------|
| `rating`   | Cijeli broj između 1 i 5 (ili null)  |
| `status`   | Mora biti `watched` ili `want_to_watch` |
| `comment`  | Maksimalno 500 znakova (mb_strlen)   |

**Razlog:** Klijentska validacija (u JavaScriptu) može se zaobići izravnim slanjem HTTP zahtjeva (npr. putem curl-a). Server mora uvijek sam provjeriti podatke jer je jedino pouzdano okruženje.

---

### 5. Provjera vlasništva (Authorization)

**Odluka:** Prije svake izmjene ili brisanja `user_movies` zapisa, controller dohvaća zapis iz baze i uspoređuje `user_id` s ID-em iz JWT tokena. Ako se ne podudaraju, vraća se 403 Forbidden.

**Razlog:** Bez ove provjere, autenticirani korisnik mogao bi izmijeniti ili obrisati tuđe zapise slanjem PUT/DELETE zahtjeva s tuđim ID-em (IDOR — Insecure Direct Object Reference).

---

### 6. Sigurnosni HTTP headeri

**Odluka:** Aplikacija postavlja sljedeće headere na svaki odgovor:

- `Content-Security-Policy: default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline';` — ograničava izvore resursa i smanjuje površinu XSS napada.
- `X-Content-Type-Options: nosniff` — sprječava MIME type sniffing koji može dovesti do izvršavanja malicioznog sadržaja.
- `X-Frame-Options: DENY` — sprječava umetanje stranice u `<iframe>` (Clickjacking zaštita).

---

### 7. Pohrana lozinki

**Odluka:** Lozinke se nikada ne pohranjuju u plain tekstu. Koristi se PHP funkcija `password_hash($password, PASSWORD_BCRYPT)` koja primjenjuje bcrypt algoritam s automatskim saltanjem. Verifikacija se vrši putem `password_verify()`.

**Razlog:** Ako napadač dobije pristup bazi podataka, bcrypt hashevi su praktično neraskidiivi zbog visoke računske složenosti i unikatnog salta po korisniku. Plain text ili slabi algoritmi (MD5, SHA1) mogu se razbiti rječničkim napadima u kratkom vremenu.

---

## Alternativne opcije koje nisu odabrane

- **Argon2id umjesto bcrypt:** Argon2id je noviji i otporniji na GPU napade, ali bcrypt je dovoljan za aplikaciju ove razine i nativno podržan u PHP-u bez dodatnih ekstenzija.
- **HttpOnly cookie za JWT umjesto localStorage:** Cookie s `HttpOnly` flagom je sigurniji od localStorage (nije dostupan JavaScriptu), ali zahtijeva implementaciju CSRF zaštite za sve API pozive. Za ovu aplikaciju odabran je localStorage uz eksplicitnu CSRF zaštitu na auth rutama.

---

## Zaključak

Implementirane sigurnosne mjere pokrivaju OWASP Top 10 rizike relevantne za ovu aplikaciju: SQL Injection (A03), XSS (A03), CSRF (A01 — broken access control), IDOR (A01) i Insecure Design (A04). Kombinacija prepared statements, output escapinga, CSRF tokena, server-side validacije i JWT autorizacije čini aplikaciju sigurnom za korištenje u razvojnom okruženju.
